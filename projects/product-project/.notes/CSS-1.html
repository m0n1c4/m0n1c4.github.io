Classes and IDs

Classes (and IDs) are just names we can apply to any HTML element. They don't have any styling information to them all by themselves. They provide us a way to reference an element by a name.

IDs are Unique

Each element can have only one ID, and each page can have only one element with that ID. Use IDs to identify a unique element.

Classes are Reusable

You can use the same class on multiple elements, and you can use multiple classes on the same element. Use classes to classify multiple objects or styles.

1
2
3
  <div class="widget"></div>
  <a href="#" class="widget"></a>
  <span class="widget"></span>

You can now use the class name .widget as your hook to apply the same set of styling to all of these, even though they are different HTML elements.

Combinations

Can you have an ID and a class on the same element? Why would you do that?

Answer

There is nothing stopping you from having both an ID and a class on a single element.

1
<div id="widget-12345" class="widget">[widget text]<div>


Here, .widget is a class for styling all widgets, and #widget-12345 is a unique ID that we can use to reference that particular widget in JavaScript, or to link directly to it as a unique URL.

Chaining

What is the difference between these two selectors?

1
.widget.red {}

1
.widget .red {}
Answer

A single space!

They look nearly identical, but the top one has no space between .widget and .red while the bottom one does. This small difference makes a huge difference in what it does. To some of you, that top selector may seem like a mistake, but it's actually quite useful.

No Space means And

1
.widget.red {}


"Select the element which has a class name of .widget and also a class name of .red."

Space means Descendant

1
.widget .red {}


"Select all elements with the class name .red that are descendents of the element .widget."


Semantics

When choosing a class or ID it is important to choose something that has value to the actual context of that element.

Content, not Style

"For example, if you have a div with an orange background that contains social media links your first inclination might be to give the div a class of .orange. What happens if that orange background is later changed to blue? Having a class of .orange. no longer makes sense. A better, more semantic, choice for a class would be .social. as it pertains to the contents of the div, not the style."

— Shay Howe, Elements & Semantics

"Class names should never map directly to a CSS property, or a given visual effect you want to achieve (such as floating, or a border effect, or a background colour, etc.). It’s shockingly bad as an ideology because it encourages a direct mapping between class name and display action. A class name, like an ID, should be descriptive in a semantic sense. ie., it should describe something about the content of the tag to which it applies. “float” is not describing content. It is describing an action you want to perform."

— Matt Wilcox, Naming and using id's and classes properly

Redundancy

1
<a href="http://css-tricks.com" class="link">CSS-Tricks.com</a>

We already know this element is a link—it's built-in to the element.

If you want to style it like a link, you can use the element <a>. If you are making a more complex site and need to distinguish between types of links, choose a better classname. For example: .search_link and .profile_link.

Using IDs

Why we don't use IDs

There can only be one element with an ID per page, so that element is not re-usable on that page.

CSS should always focus on re-usability, about creating patterns instead of one-offs. IDs are unique by nature, therefore they work inherently against reusability. No matter how much you think you’ll only ever use a pattern once, somewhere down the line you’ll find yourself needing it again.

Let's say you have a #search box in your header. Then later you/your designer/etc decides that they want search in the footer, too. So you duplicate your #search div and... everything seems cool.

It's confusing, because your CSS will still apply. However, if you're using JavaScript on that #search, it will now break, because the JavaScript doesn't know which one you mean.

It's almost always better to use reusable classes instead of IDs to prevent this potential issue. A good rule of thumb for me is that unless it's truly unique, like our #widget-12345 example, use a class.

Why we DO use IDs

Let's take our previous search example. If we used .search instead of #search, the JavaScript will most likely work on both which is almost as bad. This is why we declare uniques with IDs, and reusable elements with classes.

What if there is an element you know is going to be unique on every page? One that, if it did appear twice on the page, it would be a signal that something was wrong, e.g. you broke something in a template, or your CMS is spitting out strange chunks of code, etc.

It's a common practice to use an ID to define pages of a site, e.g. <div id="profile-page"> and <div id="signup-page">. You do this so that as a wrapper on the entire page, any elements that need to be customized can be, with that ID as a parent (that overrides all other classes), and so that you know what page you are on. Using an ID solidifies this in the code. If there happened to be two of those elements, there might be some sort of visual give-away that things aren’t working as expected.
